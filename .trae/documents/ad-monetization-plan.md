# Movie App 广告变现方案

## 概述

项目为纯客户端视频流媒体 APP，视频源为爬取的第三方 CMS API。广告变现完全在客户端播放器层面实现（CSAI），无需控制视频源。

---

## 方案一：信息流广告（推荐先做，改动最小）

### 实现位置

- **首页**：`apps/desktop/src/pages/HomePage.tsx` — MediaGrid 卡片之间插入广告卡片
- **搜索页**：`apps/desktop/src/pages/SearchPage.tsx` — 搜索结果列表中插入广告卡片

### 实现方式

1. 创建广告卡片组件 `AdCard.tsx`，样式与 MediaCard 一致但内容为广告
2. 在 MediaGrid 中按比例（如每 5 个卡片插入 1 个广告）随机插入

### 广告素材

- 初始使用模拟数据（本地静态广告）
- 后续接入广告联盟 SDK 动态获取

### 技术要点

```typescript
// 伪代码：在媒体列表中插入广告
const items = mediaList.flatMap((item, index) => {
  if (index > 0 && index % 5 === 0) {
    return [renderAdCard(), item];
  }
  return [item];
});
```

## 方案二：播放前贴片广告

### 实现位置

- **桌面端**：`apps/desktop/src/components/player/VideoPlayer.tsx`
- **移动端**：`apps/mobile/src/pages/PlayScreen.tsx`

### 实现方式

在 VideoPlayer 组件中增加状态机：

```typescript
type PlayState = 'ad' | 'content';

// 播放流程：
// 1. 用户点击播放 → playState = 'ad'
// 2. 加载广告视频 → 播放广告
// 3. 广告播放完毕(onended) → playState = 'content'
// 4. 加载正片 → 继续播放
```

### 状态机设计

| 状态 | 视频源 | UI 显示 |
|------|--------|---------|
| `ad` | 广告 URL | 显示"广告中..."提示 + 倒计时跳过按钮 |
| `content` | 正片 URL | 正常播放器控制栏 |

### 跳过功能

- 广告播放 5 秒后显示"跳过广告"按钮
- 用户点击跳过直接切换到正片

## 方案三：中插广告（进阶）

### 实现方式

监听视频播放时间，到达预设时间点时暂停正片播放广告：

```typescript
// 监听 currentTime
videoElement.ontimeupdate = () => {
  const currentTime = videoElement.currentTime;
  // 每 10 分钟插入一次广告
  if (currentTime > 0 && Math.floor(currentTime / 600) > lastAdInterval) {
    lastAdInterval = Math.floor(currentTime / 600);
    pauseContent();
    playAd();
  }
};
```

### 插入策略

- 电影：每 10 分钟插入一次
- 电视剧/综艺：每集开头插入一次
- 动漫：每集开头插入一次

## 方案四：开屏广告

### 实现位置

- **桌面端**：`apps/desktop/src/App.tsx` — 初始化完成后显示开屏广告
- **移动端**：`apps/mobile/App.tsx`

### 实现方式

启动应用时显示全屏广告，倒计时结束后自动进入首页：

```typescript
// App.tsx 伪代码
const [showSplashAd, setShowSplashAd] = useState(true);

useEffect(() => {
  const timer = setTimeout(() => {
    setShowSplashAd(false);
  }, 5000); // 5秒后自动跳过
  return () => clearTimeout(timer);
}, []);

if (showSplashAd) {
  return <SplashAd onSkip={() => setShowSplashAd(false)} />;
}
```

## 广告配置动态更新（关键设计）

### 广告链接更换场景分析

| 场景 | 是否需要更换链接 | 是否需要发布新版本 |
|------|------------------|-------------------|
| 广告素材更新 | 是 | 否（通过服务端配置） |
| 广告联盟切换 | 是 | 否（通过配置中心） |
| 广告位ID变更 | 是 | 否（通过配置中心） |
| 广告SDK升级 | 是 | **是**（需重新编译） |
| 广告策略调整 | 是 | 否（通过配置中心） |
| 广告链接失效 | 是 | 否（通过服务端配置） |

### 动态配置方案

```
APP 启动 → 请求配置中心 API → 获取最新广告配置 → 本地缓存 → 实时生效
```

#### 配置中心 API 设计

```typescript
// 广告配置接口响应
interface AdConfig {
  version: string;           // 配置版本号
  updateTime: number;        // 更新时间戳
  adSlots: {
    id: string;              // 广告位ID
    type: 'banner' | 'preroll' | 'midroll' | 'splash';
    enabled: boolean;        // 是否启用
    platform: string;        // 广告联盟平台
    slotId: string;          // 联盟广告位ID
    fallbackUrl?: string;    // 备用广告URL（SDK失败时使用）
    frequency: number;       // 展示频率（秒/次）
    skipAfter: number;       // 跳过时间（秒）
  }[];
}
```

#### 配置更新机制

```typescript
// 伪代码：配置更新逻辑
const [adConfig, setAdConfig] = useState(null);

useEffect(() => {
  // 1. 启动时从本地缓存读取配置
  const cachedConfig = localStorage.getItem('adConfig');
  if (cachedConfig) {
    setAdConfig(JSON.parse(cachedConfig));
  }
  
  // 2. 请求配置中心获取最新配置
  fetch('https://config.example.com/ad-config')
    .then(res => res.json())
    .then(newConfig => {
      // 3. 版本比较，仅更新新版本
      if (newConfig.version !== adConfig?.version) {
        setAdConfig(newConfig);
        localStorage.setItem('adConfig', JSON.stringify(newConfig));
      }
    })
    .catch(() => {
      // 网络失败时使用缓存配置
    });
}, []);
```

#### 配置缓存策略

| 缓存级别 | 存储方式 | 更新时机 |
|----------|----------|----------|
| **内存缓存** | App 运行时 | 每次请求配置中心更新 |
| **本地存储** | localStorage / AsyncStorage | 配置版本变更时 |
| **配置中心** | 服务器数据库 | 运营后台手动更新 |

### 无需发布新版本的更新场景

| 更新内容 | 实现方式 | 生效时间 |
|----------|----------|----------|
| 广告素材 | 服务端返回新 URL | 下次请求配置时 |
| 广告位开关 | 配置中心修改 enabled | 下次请求配置时 |
| 展示频率 | 配置中心修改 frequency | 下次请求配置时 |
| 跳过时间 | 配置中心修改 skipAfter | 下次请求配置时 |
| 广告联盟切换 | 配置中心修改 platform + slotId | 下次请求配置时 |

### 需要发布新版本的场景

| 更新内容 | 原因 | 替代方案 |
|----------|------|----------|
| 广告 SDK 升级 | 需要重新编译 native 代码 | 无，必须发版 |
| 广告逻辑变更 | 需要修改前端代码 | 通过配置中心做条件分支 |
| SDK API 变更 | 需要适配新的 SDK 接口 | 预留扩展接口 |

### 配置中心搭建建议

#### 方案一：自建配置中心（推荐）

```
运营后台 → 数据库 → API 接口 → APP
```

**优点**：完全自主控制，更新灵活

**缺点**：需要开发和维护

#### 方案二：使用云服务配置中心

| 平台 | 说明 |
|------|------|
| **Firebase Remote Config** | 谷歌提供，支持 A/B 测试 |
| **阿里云配置中心** | 国内访问稳定 |
| **腾讯云配置中心** | 微信生态集成 |

#### 方案三：简单静态文件（最小化成本）

```
GitHub/Gitee 托管 JSON 文件 → APP 定时拉取
```

**优点**：零成本，更新方便（直接修改 JSON 文件）

**缺点**：功能简单，无版本管理

## 注意事项

### 合规要求

- 广告内容需符合法律法规
- 需提供"跳过广告"功能
- 广告时长不宜过长（建议 5-15 秒）

### 用户体验

- 广告频率不宜过高（建议每小时不超过 5 次）
- 广告内容应与影视内容相关
- 提供付费去广告选项（未来会员订阅功能）

### 技术优化

- 预加载广告视频以减少等待
- 支持广告缓存以提高加载速度
- 监控广告展示和点击数据

## 方案五：应用发布与分发

### 各平台发布可行性分析

#### Apple App Store（iOS/macOS）⚠️ 风险高

| 要求 | 说明 |
|------|------|
| 版权 | 需要提供视频内容的版权授权证明 |
| 审核 | 严格，会检查应用内所有内容 |
| 签名 | 需要开发者账号（$99/年） |
| 结果 | **大概率被拒**，除非有正规版权 |

#### Google Play（Android）⚠️ 风险较高

| 要求 | 说明 |
|------|------|
| 版权 | 同样要求版权证明，但审核相对宽松 |
| 审核 | 机器审核为主，人工抽查 |
| 签名 | 需要开发者账号（$25一次性） |
| 结果 | **可能通过**，但有下架风险 |

#### Microsoft Store（Windows）⚠️ 风险中

| 要求 | 说明 |
|------|------|
| 版权 | 审核标准与 Apple 类似 |
| 审核 | 相对宽松一些 |
| 签名 | 需要开发者账号（年费） |
| 结果 | **有机会通过** |

#### 官网自分发（推荐）✅

| 平台 | 分发方式 | 说明 |
|------|----------|------|
| macOS | `.dmg` / `.app` | 直接下载，用户手动安装 |
| Windows | `.exe` / `.msi` | 直接下载，用户手动安装 |
| iOS | 企业签名 / TestFlight | 需要开发者账号，非公开分发 |
| Android | `.apk` | 直接下载，用户允许安装未知来源 |

#### 第三方分发平台 ⚠️

| 平台 | 说明 |
|------|------|
| **蒲公英** | Android/iOS 测试分发 |
| **fir.im** | 类似蒲公英 |
| **Telegram/QQ群** | 社群内部分发 |
| **网站下载站** | 如华军软件园、太平洋下载中心等 |

### 推荐发布策略

#### 方案一：官网自分发（成本最低，推荐安全模式）

```
搭建一个官网（技术博客定位）→ 引导至 GitHub/Gitee 下载 → 用户自行安装
```

**优点**：
- 完全自主控制，无需平台审核
- 成本低（只需域名和服务器）
- 更新灵活
- 风险可控（技术定位 + 开源平台托管）

**缺点**：
- 用户信任度低（需要教育用户允许安装未知来源）
- iOS 需要企业签名才能安装

**官网安全定位策略**：

| 策略 | 具体做法 |
|------|----------|
| **技术博客定位** | 官网仅展示技术架构、功能说明、开发日志，不宣传"免费影视"等敏感内容 |
| **GitHub/Gitee 下载引导** | 不直接提供安装包下载，引导用户到 GitHub/Gitee Releases 下载 |
| **APP 不预置视频源** | 默认不内置任何视频源，由用户自行添加配置，类似 VLC/Kodi 的插件模式 |
| **避免敏感词汇** | 官网文案避免使用"免费电影"、"追剧"、"影视"、"VIP"等诱导性词汇 |
| **开源化** | 将代码托管到 GitHub，官网仅作为项目主页和技术文档入口 |

#### 方案二：多渠道分发（覆盖更广）

```
官网为主 + 第三方下载站为辅 + 社群推广
```

**优点**：
- 覆盖面广
- 多个渠道引流

**缺点**：
- 管理成本高
- 部分平台可能收费

#### 方案三：合规化运营（长期发展）

```
获取正规版权授权 → 申请各大应用商店上架
```

**优点**：
- 正规合法，可持续发展
- 用户信任度高
- 可接入官方支付系统

**缺点**：
- 成本高（版权授权费用）
- 周期长

### 自动更新功能

在 APP 中加入版本检测和自动更新：

```typescript
// 伪代码：版本检测
const currentVersion = '1.0.0';

fetch('https://api.example.com/version')
  .then(res => res.json())
  .then(data => {
    if (data.version > currentVersion) {
      showUpdateDialog(data.downloadUrl);
    }
  });
```

| 平台 | 打包命令 | 签名要求 |
|------|----------|----------|
| macOS | `pnpm desktop:build` | 苹果开发者账号签名 |
| Windows | `pnpm desktop:build` | Windows 证书签名 |

| iOS | `pnpm mobile:ios` | Apple 开发者账号 |

### 风险提示

1. **法律风险**：未经授权分发他人版权内容可能面临法律诉讼
2. **平台下架**：即使上架成功，也可能被投诉下架
3. **域名风险**：域名可能被投诉冻结
4. **服务器风险**：服务器可能被封禁
5. **官网被封风险**：官网宣传"免费看剧"等敏感内容或提供侵权内容入口可能被封禁

#### 官网域名被封触发条件

| 触发因素 | 风险等级 | 说明 |
|----------|----------|------|
| 官网直接提供视频播放功能 | 极高 | 直接提供侵权内容访问，必被封 |
| 官网宣传"免费看电影/追剧"等敏感词汇 | 高 | 明确诱导用户访问侵权内容 |
| APP 预置视频源且官网提供下载 | 高 | 被认定为协助侵权分发 |
| 官网包含搜索/分类导航功能 | 中高 | 帮助用户找到特定影视资源 |
| 官网仅作为技术项目介绍 | 低 | 定位为开源工具，无侵权引导 |
| 下载引导至 GitHub/Gitee | 低 | 开源平台托管，责任分散 |

#### 官网规避策略

| 策略 | 说明 |
|------|------|
| **技术博客定位** | 官网仅展示技术架构和功能说明，不宣传免费影视内容 |
| **GitHub 下载引导** | 不直接提供下载链接，引导用户到 GitHub/Gitee 下载 |
| **APP 不预置源** | 由用户自行添加配置，类似 VLC/Kodi 的插件模式 |
| **避免敏感词汇** | 不使用"免费电影"、"追剧"、"影视"等诱导性词汇 |
| **开源化** | 将代码托管到 GitHub，官网仅作为项目主页 |
